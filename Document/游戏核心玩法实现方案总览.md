# 游戏核心玩法实现方案总览

## 0. 文档定位与修正说明

本文档的核心目的，是梳理游戏核心玩法常见的实现方案，以及这些方案在项目中的适用边界和落地要求。

文中提到的 ECA、FSM、HFSM、Behavior Tree、GOAP、ECS 等，都是当前游戏开发中常见的玩法逻辑组织方式。它们不是互相排斥的单选项，也不是“逻辑与表现分离”本身，而是可以用于承载核心玩法规则的不同实现模型。

当前项目形成的关键共识是：

```text
无论核心玩法最终采用 ECA、FSM/HFSM、Behavior Tree、GOAP、ECS，还是多种方案组合，
都必须满足逻辑与表现分离。
```

因此，本文档的主标题应是“游戏核心玩法实现方案总览”。“逻辑与表现分离”是贯穿所有方案的统一架构约束，而不是本文唯一讨论的实现方案。

关联文档：

- [文档迭代要求.md](文档迭代要求.md)
- [逻辑表现分离.md](逻辑表现分离.md)

## 1. 核心目标

核心玩法实现方案需要解决的不只是“代码放在哪一层”，而是以下几个长期问题：

- 玩法规则如何组织与演进。
- 状态变化如何保持确定、可追踪、可回滚。
- 表现层如何消费逻辑结果，而不反向污染逻辑。
- Replay、Undo/Redo、自动化测试、服务器校验等能力如何从架构上被支持。
- 不同复杂度、不同规模、不同性能需求的玩法，如何选择合适的逻辑实现模型。

当前推荐的总体原则可以概括为：

```text
输入数据化
规则确定化
状态修改原子化
一次操作事务化
表现结果投影化
外部副作用边界化
```

---

## 2. 核心共识

### 2.1 逻辑实现方案与架构约束不是同一件事

ECA、FSM、HFSM、Behavior Tree、GOAP、ECS 主要回答：

```text
玩法规则和决策逻辑如何组织？
```

逻辑与表现分离主要回答：

```text
谁拥有真状态？谁只负责显示、反馈和交互？
```

两者关系是：

```text
逻辑实现方案负责承载玩法规则。
逻辑与表现分离负责约束职责边界。
```

### 2.2 逻辑层是唯一真相

逻辑层负责决定游戏事实，包括：

- 胜负
- 资源
- 步数
- 位置
- CD
- 任务进度
- 关卡状态
- 生成、删除、合成、消除、碰撞、命中等规则结果
- 随机结果
- 逻辑 ID 分配状态

表现层负责玩家看到、听到、感受到的反馈，包括：

- 动画
- 特效
- 音效
- UI
- 镜头
- 飘字
- 高亮
- 过渡演出
- 对象池中的可视对象状态

表现层可以订阅逻辑结果，也可以把输入翻译成逻辑意图，但不应该直接决定或修改真状态。

### 2.3 逻辑层必须能在无表现环境下运行

核心玩法逻辑应尽量不依赖以下对象：

- GameObject
- Transform
- Renderer
- Animator
- AudioSource
- UnityEngine.Time
- UnityEngine.Random
- 具体 UI 控件
- 具体特效或动画资源

这样才能支持：

- EditMode 单元测试
- Headless Replay
- 自动化测试
- AI 离线模拟
- 服务器校验
- 多表现层替换
- 调试与回放

## 3. 常见核心玩法实现方案

| 方案 | 主要解决的问题 | 适用场景 | 注意事项 |
| --- | --- | --- | --- |
| ECA | 用 Event / Condition / Action 组织规则触发与响应 | 消除、合成、任务、关卡事件、规则连锁 | Action 不应直接操作表现，应产出状态变化、事件或操作批次 |
| FSM | 用有限状态表达明确流程 | 角色状态、关卡流程、技能阶段、简单 AI | 状态数量过多时容易膨胀 |
| HFSM | 用层级状态管理复杂流程 | 角色复合状态、Boss 行为、复杂关卡阶段 | 需要清晰划分父子状态职责 |
| Behavior Tree | 用树结构组织决策流程 | NPC AI、敌人行为、可视化 AI 编辑 | 节点执行结果应保持数据化，避免直接驱动表现对象 |
| GOAP | 用目标与规划生成动作序列 | 智能 AI、策略型 NPC、开放式行为选择 | 规划结果仍需转换成逻辑操作和表现计划 |
| ECS | 用组件和系统组织大量实体的批量处理 | 大量同构实体、性能敏感模拟、数据导向玩法 | ECS 是实现手段，不是逻辑分离的必要条件 |
| Rule Engine + OperationBatch | 用规则求解和操作批次管理复杂状态变化 | 需要 Undo/Redo、Replay、复杂连锁的中大型玩法 | 推荐作为当前项目核心玩法的基础方向之一 |
| Command + Event | 统一输入、回放、网络和调试入口 | Replay、AI 输入、网络同步、自动化测试 | 需要区分 IntentCommand、Trigger、DomainEvent、StateDiff |

这些方案可以组合使用。例如：

- 用 FSM 管理关卡主流程。
- 用 ECA 处理规则触发和连锁。
- 用 OperationBatch 收集一次操作引发的所有状态变化。
- 用 Behavior Tree 或 GOAP 生成 AI 的输入意图。
- 用 ECS 管理大量实体的状态更新。
- 用 Projection / ReadModel 把逻辑结果转换给表现层。

## 4. 推荐总体架构

当前推荐的总体数据流如下：

```text
RawInput
-> IntentCommand
-> Application / UseCase
-> RootTrigger
-> RuleEngine / ECA / FSM / BT / GOAP / ECS
-> RuleAction
-> AtomicOperation / DomainEvent / StateDiff
-> OperationBatch
-> GameState
-> CommandResult
-> Projection / Presentation / UndoManager / SaveScheduler / DebugTrace
```

推荐分层如下：

```text
Input Layer
    捕获点击、拖拽、按键、AI、Replay、Network 输入

Application / UseCase Layer
    输入门禁、流程编排、Undo/Redo、表现队列、存档边界、外部副作用协调

Logic Core
    GameState、规则求解、OperationBatch、DomainEvent、StateDiff

Projection / ReadModel Layer
    把逻辑状态和变化投影成表现/UI 友好的只读数据

Presentation Layer
    View、Animation、Effect、Audio、UI

Infrastructure Layer
    存档、网络、BI、广告、支付、资源加载、平台能力
```

依赖方向：

```text
外层可以依赖内层
内层不依赖外层
Logic Core 不依赖 UnityEngine
```

## 5. 各方案下的逻辑/表现分离方式

### 5.1 ECA

- Event 更接近逻辑内部 Trigger，不等同于对外发布的业务事件。
- Condition 必须只读状态，不写状态，不播放表现，不调用外部服务。
- Action 应产出逻辑结果，例如 AtomicOperation、DomainEvent、StateDiff 或后续 Trigger。
- 表现层根据 DomainEvent、StateDiff 或 VisualPlan 播放动画、特效、音效。

### 5.2 FSM / HFSM

- 状态机维护逻辑状态和状态转移。
- 状态进入、退出、切换时产出状态变化事件或操作批次。
- 表现层监听状态结果，映射到动画、特效、UI，而不是让 Animator 决定玩法状态。
- HFSM 的子状态可以独立计算，但仍必须遵守逻辑层不依赖表现层的约束。

### 5.3 Behavior Tree

- BT 节点负责决策，输出意图、动作结果或黑板数据变化。
- 黑板数据应属于逻辑数据，不应保存 GameObject、Transform、Animator 等表现引用。
- 表现层根据逻辑结果构建动画或特效播放计划。
- 无表现环境下，BT 仍应能完成决策与状态更新。

### 5.4 GOAP

- GOAP 负责根据目标和世界状态生成计划。
- 计划本身不是表现，它需要转换成逻辑可执行的 IntentCommand、RuleAction 或操作序列。
- 计划执行产生的状态变化进入 OperationBatch。
- 表现层只消费执行结果，不参与规划判定。

### 5.5 ECS

- 逻辑组件保存真状态，例如位置、属性、AI 数据、任务状态。
- 表现组件保存模型、动画、粒子、UI 映射等可视数据。
- 逻辑系统只操作逻辑组件。
- 表现系统根据逻辑状态、事件或快照更新表现组件。
- 无表现环境下，可以只运行逻辑 World 或逻辑 System。

## 6. Replay / Undo / Redo 的架构要求

Replay 和 Undo/Redo 容易被混在一起，但它们的本质不同。

```text
Replay 重放的是输入和确定性逻辑。
Undo/Redo 回滚或重做的是已经产生的状态变化。
```

因此，核心玩法实现需要提前约束：

- 输入必须数据化，形成可序列化的 IntentCommand。
- 规则求解必须确定，避免依赖系统时间、不稳定遍历顺序和未记录的随机数。
- 一次外部意图引发的所有变化应收进同一个 OperationBatch。
- AtomicOperation 需要保存 Revert 所需的数据。
- DomainEvent 表示业务事实，StateDiff 表示具体状态差异，二者服务不同目的。
- 表现层的异步动画不应改变逻辑事务边界。

## 7. 当前推荐取舍

如果对象数量不大，但规则复杂、连锁多、需要 Replay 或 Undo/Redo，优先考虑：

```text
RuleEngine / ECA + OperationBatch + DomainEvent + StateDiff
```

如果对象数量巨大、规则同构、性能瓶颈明确，再考虑以 ECS 作为核心数据组织方式。

如果主要问题是角色、关卡或技能流程管理，可以引入 FSM / HFSM。

如果主要问题是 AI 决策，可以引入 Behavior Tree 或 GOAP，但它们的输出仍应回到统一的输入、规则和状态修改链路中。

当前项目不建议把复杂玩法核心规则直接建立在 MVVM / MVP 上。MVVM / MVP 更适合 Meta UI、商店、背包、任务面板等表现和界面组织，不应取代玩法 Logic Core。

## 8. 架构图说明

配套图片“游戏核心玩法实现方案总览.png”应表达以下关系：

- 顶层是“游戏核心玩法实现方案总览”，不是单一的“逻辑表现分离”主题。
- ECA、FSM、HFSM、Behavior Tree、GOAP、ECS 是可选或可组合的逻辑实现方式。
- Logic Core 负责规则、状态、决策、事件、操作批次等核心玩法逻辑。
- 逻辑/表现接口层负责事件、消息、命令缓冲、快照、同步和表现计划。
- Presentation Layer 可替换、可缺省，可接入 Unity、Unreal、自研引擎、服务器无渲染、命令行测试工具等不同运行环境。
- 逻辑与表现分离作为设计要点贯穿全图，是所有实现方案共同遵守的约束。

## 9. 修正后的结论

本次修正主要澄清以下表述：

- 文档主旨是“游戏核心玩法实现方案总览”。
- “逻辑与表现分离”不是文件名对应的唯一主题，而是所有核心玩法实现方案都必须满足的架构原则。
- ECA、FSM、HFSM、Behavior Tree、GOAP、ECS 等内容本身是正确的，应保留为常见实现方案说明。
- 需要避免把“选择哪种逻辑实现方案”和“是否需要逻辑/表现分离”混为一谈。

最终结论：

```text
核心玩法可以采用不同逻辑实现方案。
不同方案可以组合。
但真状态必须归逻辑层，表现层只消费逻辑结果。
逻辑层必须支持在无表现环境下独立运行。
```
